钉钉新CEO陈宇森称「未来的软件是日抛式的」，如何看待此观点？

不要绝对地去理解「未来的软件是日抛式的」这句话。

如果把它当作一种非黑即白的预言，认为所有经典的大型软件都会消亡，那未免有些哗众取宠。但如果我们把这句话翻译成——未来会涌现出海量“用完即弃”的按需软件，那么陈宇森的观点不仅是正确的，甚至可以说是技术发展的必然。

被 AI 赋能的“超级脚本”
其实作为程序员，“日抛式”软件对我们来说并不是什么新鲜概念。

在日常的开发、架构设计或系统维护中，我们本来就会高频地编写大量一次性的代码。无论是跑个临时数据、处理个自动化任务，还是洗一遍日志，这些随手写就、用完就丢的几百行代码，我们在行话里统称为“脚本”。在这个意义上，日抛式的程序一直存在，只是过去它被封印在终端的命令行里，是少数技术开发者的专属工具。

真正发生改变的，是 AI 的介入打破了这层技术壁垒。

现在，只要需求描述得足够明确，AI 就能以远超人类手写代码的效率，瞬间帮你生成这些一次性的应用。更具有颠覆性的是，过去程序员随手写的脚本往往简陋且没有图形界面，而今天 AI 随手写出来的软件，不仅可以拥有完整的 UI 交互，还能承载相当复杂的业务逻辑。当“写代码”的边际成本趋近于零时，软件形态走向“用完即弃”就是一种经济学上的必然。

真正的胜负手在于“场景”
既然技术底座的演进已经没有悬念，那么讨论的焦点就应该转移到更核心的商业与产品逻辑上：用户究竟会在什么样的场景下，去召唤并使用这些“日抛式”的软件？

陈宇森抛出这个观点，显然是想为钉钉正在大力推进的智能体生态造势。钉钉试图在办公和企业协作的场景下去跑通这个逻辑，通过 AI 随用随写，解决企业内部大量长尾的、碎片化的定制审批或协同需求。只要阿里愿意持续在这个方向上投入资源去试错，这绝对是一个顺理成章的探索方向。

为什么更具想象力的是微信？
但如果我们把视线放宽，跳出企业服务的局限，我会觉得在 ToC（面向普通消费者）的场景下，日抛式软件会爆发出更惊人的能量，而最有可能把这件事做成并做透的，其实是微信。

不妨想象这样一个未来场景：在微信的交互界面里，你直接对着腾讯的某个超级智能体说出你的临时需求。这个智能体不需要你去下载任何新的 App，而是直接在微信庞大且成熟的底层生态下，为你实时生成并拉起一个“专属小程序”。

这个即用即抛的小程序，不仅能完美契合你当下的特定痛点，还能极其顺滑地调用微信支付、地理位置、甚至社交关系链等底层接口，帮你完成一次个性化的交易或自动化服务。服务结束，程序消散，不占内存，不留痕迹。

钉钉试图用日抛软件去解决企业效率的边角料问题；而基于国民级应用的日抛软件，重塑的将是普通人与数字世界交互的根本路径。陈宇森确实敏锐地点出了未来软件演进的趋势，但这个趋势最终结出的最大果实，未必会长在钉钉之上。

我个人倒是觉得一半甚至更少是需求定制型的日抛软件，另外的都是现在的已有软件。毕竟需求普遍化，每日都用的话，必然会有一个巨大市场，也会有巨大的习惯依赖度保证效率。只有小众需求，定制化需求，每天变化的需求才会日抛，份额注定很少。

上面是我对一个知乎问题的回答以及跟网友的讨论。我想再补充一下：每个企业、中小企业，以及每个人对于软件或各种产品的需求，其实都是很个性化的。看过去的年代，受限于生产力，更多是一些统一的产品提供给全世界的人使用。其中最典型的代表就是苹果手机，每一款苹果手机的销量都是过千万甚至过亿的，大家用的至少在外观上一模一样的苹果手机。这种统一的根源是受制于供应链和硬件生产能力的限制。可能在渴望的将来，如果 3D 打印以及其他纺织、组装技术没法得到更加革命性、突破性的发展，那么这种硬件上的统一可能还是会长期存在。但是软件方面，我认为 AI 的出现就会有很大影响了。过去我们所有人都是使用同样的操作系统，使用同样的办公软件，使用同样的即时聊天工具（比如微信）。这仅仅是一个生活方式而已，这个生活方式大家都需要一样吗？生活应该是多姿多彩的，应该是有很多个生活-方式的。随着 AI 的出现，我认为会有更多个性化的软件出现。每个人自己通过 AI 的使用，适时地生产出来，仅供自己或者少数几个人、自己的公司、自己的团队内部使用。因为这样能够最贴合自己的使用场景，它必然是最高效的。我昨天倒是遇到过一个很实际的例子。我日常需要发表公众号文章，即便我是使用 AI 去帮我代笔，但在最终发表的时候，实际上还是需要对文章去做各种-样的编辑、修改、润色，以确保是我想要表达的文字。在过去这个过程一直很麻烦：1. 我的底稿是在本地写的，写完之后可能要复制粘贴去微信公众号的编辑器，重新做一下排版，做一下最终的修改。2. 很多时候我还会把文章先通过预览发送给我一些朋友看，根据朋友的意见再进行--进行修改。3. 这一层的修改对于我来说，最方便的其实是在微信编辑器里面去修改。4. 这样一来，微信里面的版本跟我最终发布出去的版本，跟我本地原来的底稿版本它就会不一样了，有的时候改变其实还挺大的。5. 我本地的底稿是使用 Markdown 格式来保存的，其实是可以使用 Obsidian 去处理，但是我要如何把这个修改给同步到 Obsidian 本地的目录呢？6. 我之前甚至写过程序去调用微信公众号的后台后端 API，来实现一个双向同步。但那个程序我写完之后也是很少使用，因为太麻烦了。最方便的方式其实就是在我完成编辑发送之后，在预览页面直接复制整篇文章，粘贴到本地。因为微信公众号里面是 HTML 格式的，我需要有个程序自动地帮我把这个 HTML 内容复制粘贴到本地，并自动帮我转为 Markdown 格式，这样我就可以保持文章的主体内容，直接粘贴到 Obsidian 里面。但是 Obsidian 的 Markdown 语法识别有一些 bug，主要是我在分点展示列表的时候，它对列表的处理有 bug。而且很多时候我会觉得 Obsidian 这个软件对我来说也太重了，我并不需要预览，我只需要一个完整的 Markdown 直接显示，然后我就可以编辑了，这样最快。所以我日常最常使用 Sublime Text 来做这些文档处理。而我从网页里面复制 HTML 再粘贴到 Sublime Text 里面的话，它会变成纯文本，就连 Markdown 格式，也不会支持。昨天晚上我心血来潮想要解决一下这个问题，了解一下为什么这种复制粘贴在不同的软件里面不一样，具体是什么原因。于是我就去跟 AI 聊天，了解了背后具体的技术细节之后，我发现其实是可以自己开发一个 Sublime Text 的插件。这样我在 Sublime 里面进行粘贴的时候，也可以粘贴为 Markdown 的格式。我就打开了 Codex，直接跟 ChatGPT 讲说我要开发这么个插件。几分钟它就全自动地帮我把这个插件给实现了。然后使用并基本上调试一下，只遇到了一个小问题。跟 AI 反馈一下之后，它也就立刻改好了。既然是插件，我也可以发布去 Semantic Text 的插件市场，让更多人可以使用。这需要有一定流程的，我以前做过这样的流程，其实还是有点麻烦的。但是我发现，我把这个需求跟相应的操作权限给了 GitHub 和
