**我说句得罪人的话：大多数人现在的 AI 编程方式，可能都走偏了。**

当大家还在吹捧 Cursor 怎么好用，还在沉迷 VS Code 里的各种 Copilot 插件时，我做了一个让很多一线开发者觉得“疯了”的决定：

**我把 IDE 全卸了。**

Cursor？不用。VS Code？不装。

我现在写代码的工具链，干净得甚至有点“简陋”：

- **Sublime Text：** 纯文本模式，只读。
    
- **Fork：** 看 Git 提交，只读。
    
- **Terminal：** **这是我唯一的输入接口。**
    

日常开发，我轮番轰炸 Claude Code、Codex、Gemini CLI，但我绝不碰一行代码。

**为什么？因为在 AI 时代，抱着编辑器写代码，是在用战术上的勤奋，掩盖战略上的懒惰。**

### 所谓的“编程”，本质就是“管理”

我在终端里敲下每一行 Prompt 的感觉，跟我当年做 CTO 时简直一模一样。

回想一下，当你坐到技术总监、CTO 这个位置时，你还会亲自去写 `if (x == y)` 吗？

**绝不可能。**

那时候我的工作是什么？

是翻看代码库（Sublime），是审查提交记录（Fork），是听取需求，然后对着团队**“指指点点”**。

是的，就是“指指点点”。这个词听起来刺耳，但却是技术管理的精髓。

我不需要知道这行代码是你用左手敲的还是右手敲的，我只关心：

1. **意图传达下去没有？**
    
2. **团队理解对了没有？**
    
3. **最后交付的结果（Acceptance），达标没有？**
    

过去，接指令去改代码的是坐在我对面的“小张”和“小李”；

**现在，接指令的是 Claude 和 Gemini。**

我和 AI 的关系，不是“人与工具”，而是“经理与员工”。

这种面向 Agent 的开发流程，根本不需要 IDE。一旦你打开了 IDE，你就忍不住想去修修补补，你就忍不住降级成了一个“搬砖工”。

### Cursor 很好，但它把你困在了“砖头”里

为什么我看不上 Cursor 这种工具？

因为它们做得太好了——好到让你舒服地陷在**“一线编码”**的泥潭里出不来。

Cursor 的逻辑依然是：如何更快地把这行代码补全？如何更优雅地重构这个函数？

**格局小了。**

它还在教你如何把每一块砖搬得更直、更漂亮。但 AI 时代，算力过剩，代码生成极其廉价。

**哪怕这块砖搬歪了又怎样？让 AI 砸了重搬就是了，一秒钟的事。**

如果你还在纠结单行代码的颗粒度，你就是在浪费生命。

我们应该关心的是什么？

是**园区规划**，是**摩天大楼的蓝图**，是整个系统的**顶层设计**。

### 哪怕你只有一个人，也要像个 CTO 一样战斗

很多人说，我只是个独立开发者，我没有团队。

**错，AI 就是你的团队。**

我现在虽然是一个人对着屏幕，但我感觉背后站着一支 24 小时待命的精锐连队。

我只负责在终端里发号施令：

- “重构这块逻辑。”
    
- “在这个模块加个接口。”
    
- “刚才的改动有问题，Review 一下 diff，给我回滚重做。”
    

这才是 AI 时代的正确打开方式。

**别再跟 AI 比拼打字速度了，去比拼你的审美、你的判断力、你的架构能力。**

甚至可以说，如果你现在还在用 IDE 逐行 Review AI 写的代码，那你其实是在**给 AI 打工**。

**跳出来吧。**

关掉 VS Code，打开终端。

别做那个低头搬砖的人，试着做一次那个指点江山的“包工头”。
